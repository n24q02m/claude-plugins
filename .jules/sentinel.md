## 2026-03-27 - [MEDIUM] Insecure Shell Execution Mode and Non-Portable Emptiness Checks
**Vulnerability:** Shell scripts lacked `set -euo pipefail` causing potential unbounded variable expansions, ignored pipe errors, and silently failing commands, leading to unpredictable or insecure states. Also, `[ "$(ls -A "$dir" 2>/dev/null)" ]` was used to check for directory emptiness, which can cause unexpected behavior depending on `ls` aliases and parsing.
**Learning:** For predictable execution and avoiding security risks due to unbound variables or hidden pipe failures, shell scripts must fail securely on the first error. Standard checking with `ls` is brittle.
**Prevention:** Always enforce strict execution mode by adding `set -euo pipefail` at the top of all shell scripts. To check if a directory is not empty in a POSIX/portable way, use the pattern: `[ -n "$(find "$dir" -mindepth 1 2>/dev/null | head -n 1)" ]` to avoid alias/parsing issues and improve portability.
## 2026-03-31 - [CRITICAL] Remote execution in MCP plugins handling sensitive credentials
**Vulnerability:** The MCP server configuration for `better-notion-mcp` (`plugins/better-notion-mcp/.claude-plugin/plugin.json`) used a remote HTTP server (`"type": "http"`) for execution, even though the plugin handles sensitive credentials like `NOTION_TOKEN`. This presents a massive data exfiltration risk, as the user's private data or tokens could be logged, intercepted, or stolen by the third-party endpoint.
**Learning:** Configurations for AI coding agent plugins must never use remote execution for components that integrate with authenticated or private services unless explicitly desired. The design assumption must be that plugins operate locally.
**Prevention:** To prevent data exfiltration vulnerabilities, ensure that MCP server configurations handling sensitive credentials use local command execution (e.g., `"command": "npx"` or `"uvx"`) rather than remote HTTP execution (`"type": "http"`).
