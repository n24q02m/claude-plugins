## 2026-03-27 - [MEDIUM] Insecure Shell Execution Mode and Non-Portable Emptiness Checks
**Vulnerability:** Shell scripts lacked `set -euo pipefail` causing potential unbounded variable expansions, ignored pipe errors, and silently failing commands, leading to unpredictable or insecure states. Also, `[ "$(ls -A "$dir" 2>/dev/null)" ]` was used to check for directory emptiness, which can cause unexpected behavior depending on `ls` aliases and parsing.
**Learning:** For predictable execution and avoiding security risks due to unbound variables or hidden pipe failures, shell scripts must fail securely on the first error. Standard checking with `ls` is brittle.
**Prevention:** Always enforce strict execution mode by adding `set -euo pipefail` at the top of all shell scripts. To check if a directory is not empty in a POSIX/portable way, use the pattern: `[ -n "$(find "$dir" -mindepth 1 2>/dev/null | head -n 1)" ]` to avoid alias/parsing issues and improve portability.
## 2026-04-03 - [HIGH] Output/Env Injection in GitHub Actions
**Vulnerability:** Workflow scripts were directly writing variables that contained untrusted user input (e.g., from `github.event.client_payload` and `secrets`) to `$GITHUB_OUTPUT` and `$GITHUB_ENV` using `echo "VAR=$VALUE" >> $GITHUB_OUTPUT`. Also, commit messages used potentially untrusted shell variables like `$REPO` or `$VERSION` inside strings without proper sanitization. Furthermore, not all `run` blocks explicitly enforced `set -euo pipefail`.
**Learning:** If the untrusted input contains a newline or special characters, an attacker can inject arbitrary environment variables or command parameters. Additionally, without strict execution mode, workflows may silently fail or have unbound variable vulnerabilities.
**Prevention:** Always use dynamic heredoc syntax (e.g., `EOF=$(dd if=/dev/urandom bs=15 count=1 status=none | base64)`) to securely write multi-line variables to `$GITHUB_OUTPUT` or `$GITHUB_ENV`. Map context variables to step `env` blocks and pass them to CLI tools via explicitly defined env variables (like `COMMIT_MSG`) instead of interpolation. Add `set -euo pipefail` to all `run` blocks to ensure secure failure states.
## 2026-04-08 - [CRITICAL] Command Argument Injection Vulnerability via Plugin Name
**Vulnerability:** Plugin names derived from `.claude-plugin/marketplace.json` were passed directly into the `--repo` argument for `gh release view` without validation, leading to potential command argument injection if a malicious name (e.g. `n24q02m/-foo`) or shell expansion string was used. This also applies when evaluating paths.
**Learning:** External tools (like `gh` or shell scripts) can misinterpret arguments derived from JSON configuration files when the input is malformed, leading to potential exploits. Always sanitize and validate variables before using them as arguments to shell commands or subprocesses.
**Prevention:** Strictly validate input formats using regex (e.g., `^[a-zA-Z0-9_-]+$`) in Python scripts like `scripts/validate_marketplace.py` and `scripts/check_version_freshness.py` to ensure external tool arguments and paths cannot be used to inject arguments.
## 2026-04-10 - [HIGH] Shell Command Option Injection via `echo`
**Vulnerability:** The workflow file `.github/workflows/ci.yml` used `echo "${SMTP_CREDENTIAL%%:*}" >> "$GITHUB_ENV"` to extract parts of a secret and write it to the environment. If the parsed string (the secret) started with a hyphen, `echo` could incorrectly evaluate it as an option (e.g. `-e` or `-n`) rather than a literal string, potentially exposing or misinterpreting the secret.
**Learning:** `echo` is notoriously unreliable when dealing with arbitrary, potentially untrusted variables because variables prefixed with `-` can be interpreted as flags. This applies everywhere in shell scripting.
**Prevention:** Use `printf "%s\n" "$VAR"` instead of `echo "$VAR"`. `printf` separates the format string from the arguments, rendering argument injection into options impossible.
