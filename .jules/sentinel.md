## 2026-03-27 - [MEDIUM] Insecure Shell Execution Mode and Non-Portable Emptiness Checks
**Vulnerability:** Shell scripts lacked `set -euo pipefail` causing potential unbounded variable expansions, ignored pipe errors, and silently failing commands, leading to unpredictable or insecure states. Also, `[ "$(ls -A "$dir" 2>/dev/null)" ]` was used to check for directory emptiness, which can cause unexpected behavior depending on `ls` aliases and parsing.
**Learning:** For predictable execution and avoiding security risks due to unbound variables or hidden pipe failures, shell scripts must fail securely on the first error. Standard checking with `ls` is brittle.
**Prevention:** Always enforce strict execution mode by adding `set -euo pipefail` at the top of all shell scripts. **Vulnerability:** Untrusted inputs from `repository_dispatch` and `workflow_dispatch` were directly interpolated into a `run` block and written to `GITHUB_OUTPUT` using simple `echo "key=value"` syntax.
**Learning:** Simple string interpolation (${{ ... }}) in shell scripts is dangerous as it allows command injection. Furthermore, writing to `GITHUB_OUTPUT` without heredocs allows output injection if the value contains newlines or other special characters.
**Prevention:** Always map untrusted expressions to environment variables before using them in `run` blocks. Use heredocs with unique, randomly generated delimiters when writing multi-line or untrusted data to `GITHUB_OUTPUT`.
