## 2026-03-27 - [MEDIUM] Insecure Shell Execution Mode and Non-Portable Emptiness Checks
**Vulnerability:** Shell scripts lacked `set -euo pipefail` causing potential unbounded variable expansions, ignored pipe errors, and silently failing commands, leading to unpredictable or insecure states. Also, `[ "$(ls -A "$dir" 2>/dev/null)" ]` was used to check for directory emptiness, which can cause unexpected behavior depending on `ls` aliases and parsing.
**Learning:** For predictable execution and avoiding security risks due to unbound variables or hidden pipe failures, shell scripts must fail securely on the first error. Standard checking with `ls` is brittle.
**Prevention:** Always enforce strict execution mode by adding `set -euo pipefail` at the top of all shell scripts. To check if a directory is not empty in a POSIX/portable way, use the pattern: `[ -n "$(find "$dir" -mindepth 1 2>/dev/null | head -n 1)" ]` to avoid alias/parsing issues and improve portability.
## 2026-03-27 - [HIGH] Command Injection and Output Parameter Injection in GitHub Actions
**Vulnerability:** Untrusted inputs from `repository_dispatch` and manual triggers were directly interpolated into GitHub Actions `run` blocks using expression syntax (`${{ ... }}`), allowing for command injection. Additionally, these inputs were written to `$GITHUB_OUTPUT` using simple `echo "key=value"`, which is vulnerable to Output Parameter Injection if the input contains newlines.
**Learning:** GitHub Actions `run` scripts are templates; direct interpolation of untrusted strings is equivalent to code injection. `$GITHUB_OUTPUT` also has a specific syntax for multi-line values to prevent injection.
**Prevention:** Always map untrusted context variables (`github.event.client_payload`, `inputs`, etc.) to environment variables in an `env` block and reference them as shell variables (e.g., `"$VAR"`). Use multiline heredoc syntax with a random delimiter when writing potentially multi-line values to `$GITHUB_OUTPUT`.
